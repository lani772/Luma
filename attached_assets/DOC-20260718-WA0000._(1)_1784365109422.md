## 2. Identity & User Model

### 2.1 User Record

```sql
CREATE TABLE users (
    id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email             CITEXT NOT NULL,           -- case-insensitive unique
    username          TEXT UNIQUE,               -- 3-20 chars, [a-zA-Z0-9_]
    password_hash     TEXT NOT NULL,             -- bcrypt default cost
    full_name         TEXT NOT NULL DEFAULT '',
    role              TEXT NOT NULL DEFAULT 'owner'
                      CHECK (role IN ('owner','admin','member','viewer')),
    status            TEXT NOT NULL DEFAULT 'active'
                      CHECK (status IN ('active','suspended','deleted')),
    email_verified_at TIMESTAMPTZ,
    subscription_tier TEXT NOT NULL DEFAULT 'free',
    preferences       JSONB NOT NULL DEFAULT '{}',
    created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

**Key design decisions:**
- `email` uses PostgreSQL `CITEXT` extension — `User@example.com` and `user@example.com` are the same address at the DB level, no application-layer normalisation needed.
- `username` is nullable. It was added in migration `000005_username_auth`. All users registered before that migration have `NULL` username; they log in by email only until they set one.
- `role` is a global account role (used for coarse platform-level gating). Device-level permissions use `device_admins` and the planned `permissions` table — the two systems are independent.
- `status = 'deleted'` is a soft delete. A hard `DELETE` is blocked by `ON DELETE RESTRICT` on `devices.owner_id` while the user still owns devices.

### 2.2 Go Model

```go
// internal/models/user.go
type User struct {
    ID               uuid.UUID
    Email            string
    Username         *string    // nil = not yet set
    PasswordHash     string
    FullName         string
    Role             UserRole   // owner | admin | member | viewer
    Status           UserStatus // active | suspended | deleted
    EmailVerifiedAt  *time.Time // nil = not verified
    SubscriptionTier string
    Preferences      JSONMap
    CreatedAt        time.Time
    UpdatedAt        time.Time
}
```

### 2.3 Platform-Specific Phone Record

Every login or registration creates or reuses a `user_phones` row keyed by `(user_id, device_name, platform)`. This is how the system supports multiple simultaneous sessions — each physical device gets its own push token and session lineage.

```sql
CREATE TABLE user_phones (
    id            UUID PRIMARY KEY,
    user_id       UUID REFERENCES users(id) ON DELETE CASCADE,
    device_name   TEXT NOT NULL,
    platform      TEXT CHECK (platform IN ('ios','android','web','other')),
    push_token    TEXT,          -- FCM or APNs token for push notifications
    push_provider TEXT CHECK (push_provider IN ('fcm','apns', NULL)),
    app_version   TEXT,
    last_seen_at  TIMESTAMPTZ,
    created_at    TIMESTAMPTZ,
    revoked_at    TIMESTAMPTZ    -- NULL = active phone
);
```

`FindOrCreatePhone` deduplicates by `(user_id, device_name, platform, revoked_at IS NULL)` — logging in twice from the same phone updates `last_seen_at` rather than creating a second row.

## 3. Authentication Flows

### 3.1 Token Architecture

| Token | Type | Storage | TTL | Purpose |
|---|---|---|---|---|
| Access token | HS256 JWT | Mobile AsyncStorage | 15 min | API authorisation on every request |
| Refresh token | Opaque random (32 bytes, URL-safe base64) | Mobile AsyncStorage + DB as SHA-256 hash | 30 days | Obtain a new access token without re-login |

**Why opaque refresh tokens (not JWTs)?** Opaque tokens can be revoked immediately by deleting the DB row. A JWT refresh token would remain valid until expiry even after a logout or password reset.

**JWT Claims:**

```json
{
  "sub": "<user-uuid>",
  "sid": "<session-uuid>",
  "role": "owner",
  "iss": "luma-cloud-backend",
  "exp": 1784355118,
  "iat": 1784354218
}
```

`sid` (session ID) is checked against the in-memory/Redis blacklist on every authenticated request, so a session revoked at logout takes effect within the current access-token TTL (≤ 15 min) even without rotating the token.

### 3.2 Registration

```
Mobile                       Backend
  │                               │
  │  POST /cloud/auth/register    │
  │  { email, username,           │
  │    password, fullName,        │
  │    deviceName, platform }     │
  │ ──────────────────────────►  │
  │                               │  1. Verify email uniqueness (CITEXT)
  │                               │  2. Verify username uniqueness
  │                               │  3. bcrypt hash password
  │                               │  4. INSERT users
  │                               │  5. FindOrCreatePhone
  │                               │  6. Mint refresh token (opaque)
  │                               │  7. INSERT session
  │                               │  8. Mint access token (JWT)
  │                               │  9. Audit: user.registered
  │  ◄────────────────────────── │
  │  AuthResponse {               │
  │    accessToken, refreshToken, │
  │    user { id, email,          │
  │      username, fullName,      │
  │      role, emailVerified },   │
  │    sessionId                  │
  │  }                            │
```

**Validation rules on `RegisterRequest`:**
- `email` — valid RFC 5322 email (Gin `email` validator)
- `username` — 3–20 characters, `^[a-zA-Z0-9_]+$` (custom `username` validator registered at router init)
- `password` — minimum 8 characters
- `fullName` — required, any string
- `deviceName` — required (e.g. "iPhone 15", "LUMA Mobile App")
- `platform` — `oneof=ios android web other`

### 3.3 Login

Login accepts **either** `email` **or** `username` — not both.

```
Mobile                           Backend
  │                                   │
  │  POST /cloud/auth/login           │
  │  { email OR username,             │
  │    password, deviceName,          │
  │    platform, pushToken? }         │
  │ ──────────────────────────────►  │
  │                                   │  1. Resolve user by email or username
  │                                   │  2. Reject if status != 'active'
  │                                   │  3. bcrypt.Compare(password, hash)
  │                                   │  4. FindOrCreatePhone
  │                                   │  5. Mint refresh + access tokens
  │                                   │  6. Audit: user.login
  │  ◄────────────────────────────── │
  │  AuthResponse (same as register)  │
```

Mobile routing logic in `cloud-api.ts`:
```typescript
const isEmail = identifier.includes("@");
body = isEmail ? { email: identifier } : { username: identifier };
```

### 3.4 Token Refresh (Rotation)

Refresh tokens are **single-use**. Every call to `/auth/refresh` revokes the old session and creates a new one. A replayed stolen refresh token is detectable because its hash no longer maps to an active session.

```
Mobile                           Backend
  │                                   │
  │  POST /cloud/auth/refresh         │
  │  { refreshToken }                 │
  │ ──────────────────────────────►  │
  │                                   │  1. Hash incoming token
  │                                   │  2. Lookup session by hash
  │                                   │  3. Verify not revoked, not expired
  │                                   │  4. Revoke old session + blacklist SID
  │                                   │  5. Create new session (same phone)
  │                                   │  6. Mint new access + refresh tokens
  │  ◄────────────────────────────── │
  │  New AuthResponse                 │
```

The mobile client serialises concurrent refresh calls using a module-level waiter queue (`_refreshWaiters`) to prevent thundering-herd races when multiple in-flight requests all 401 simultaneously.

### 3.5 Logout

```
POST /cloud/auth/logout
{ refreshToken }
```

1. Find session by refresh token hash.
2. Set `sessions.revoked_at = now()`.
3. Add session ID to blacklist cache (TTL = remaining access token TTL).
4. Returns 200 — idempotent (already-revoked tokens silently succeed).

## 4. Session & Multi-Device Management

### 4.1 Session Record

```sql
CREATE TABLE sessions (
    id                UUID PRIMARY KEY,
    user_id           UUID REFERENCES users(id) ON DELETE CASCADE,
    phone_id          UUID REFERENCES user_phones(id) ON DELETE SET NULL,
    refresh_token_hash TEXT NOT NULL UNIQUE,  -- SHA-256 of raw token
    user_agent        TEXT,
    ip_address        INET,
    expires_at        TIMESTAMPTZ NOT NULL,
    revoked_at        TIMESTAMPTZ,           -- NULL = active
    created_at        TIMESTAMPTZ NOT NULL
);
```

### 4.2 Session Lifecycle

```
Register / Login
    │
    ▼
session created (phone_id linked)
    │
    ├─── Normal use ──────────────────────►  access token used until expiry
    │         │
    │         ▼ access token expires
    │    POST /auth/refresh
    │         │
    │         ▼ token rotation
    │    old session revoked + new session created
    │
    ├─── Explicit logout ─────────────────►  session revoked
    │
    ├─── Password reset ──────────────────►  ALL sessions for user revoked
    │
    ├─── "Log out other devices" ─────────►  all sessions except current revoked
    │
    └─── Account deleted ─────────────────►  ALL sessions for user revoked
```

### 4.3 Session API (Auth Engine)

| Method | Path | Auth | Description |
|---|---|---|---|
| `GET` | `/cloud/auth/sessions` | Required | List all active sessions for authenticated user |
| `DELETE` | `/cloud/auth/sessions/:sessionId` | Required | Revoke a specific session |
| `POST` | `/cloud/auth/sessions/revoke-others` | Required | Revoke all sessions except current |

**`GET /auth/sessions` response:**

```json
[
  {
    "id": "uuid",
    "deviceName": "iPhone 15",
    "platform": "ios",
    "createdAt": "2026-07-18T...",
    "lastSeenAt": "2026-07-18T...",
    "isCurrent": true
  }
]
```

### 4.4 Phone Management (Users Engine)

| Method | Path | Auth | Description |
|---|---|---|---|
| `GET` | `/cloud/users/me/phones` | Required | List registered phones |
| `DELETE` | `/cloud/users/me/phones/:phoneId` | Required | Revoke phone + all its sessions |

Revoking a phone (`DELETE /me/phones/:phoneId`) revokes every `sessions` row with that `phone_id` before setting `user_phones.revoked_at`. This is the "remotely sign out a stolen phone" action.

## 5. Password & Email Management

### 5.1 Password Reset Flow

```
Mobile                           Backend
  │                                   │
  │  POST /cloud/auth/               │
  │    request-password-reset        │
  │  { email }                        │
  │ ──────────────────────────────►  │
  │  200 OK (always, no leak)         │  1. If user found: generate token
  │                                   │  2. Store hash in password_reset_tokens
  │                                   │  3. TTL: 1 hour
  │                                   │  4. Delivery via Notification Engine
  │                                   │     (currently logged, email TBD)
  │                                   │
  │  POST /cloud/auth/reset-password  │
  │  { token, newPassword }           │
  │ ──────────────────────────────►  │
  │                                   │  1. Hash token, lookup record
  │                                   │  2. Verify not used, not expired
  │                                   │  3. bcrypt new password
  │                                   │  4. Mark token used_at
  │                                   │  5. Revoke ALL sessions for user
  │  200 OK                           │
```

> **Security note:** `RequestPasswordReset` always returns 200 regardless of whether the email exists. This prevents email enumeration attacks.

### 5.2 Change Password (Authenticated)

```
POST /cloud/auth/change-password
Authorization: Bearer <access_token>
{ currentPassword, newPassword }
```

Verifies `currentPassword` against stored hash before updating. Does **not** revoke sessions (user stays logged in on all devices — consistent with most auth best practices; add session revocation here if stricter policy is required).

### 5.3 Email Verification

```
POST /cloud/auth/resend-verification    → generates EmailVerificationToken (24h TTL)
POST /cloud/auth/verify-email           → { token } → marks email_verified_at
```

`EmailVerifiedAt = nil` means unverified. The app shows the `/verify-email` screen post-registration but does **not** block API access — email verification is informational, not a hard gate.

```sql
CREATE TABLE email_verification_tokens (
    id          UUID PRIMARY KEY,
    user_id     UUID REFERENCES users(id) ON DELETE CASCADE,
    token_hash  TEXT NOT NULL UNIQUE,
    expires_at  TIMESTAMPTZ NOT NULL,
    used_at     TIMESTAMPTZ    -- NULL = unused
);
```

### 5.4 Account Deletion

```
DELETE /cloud/users/me
Authorization: Bearer <access_token>
{ password? }   -- optional confirmation
```

1. If `password` supplied: verify against stored hash, return 401 on mismatch.
2. Revoke all active sessions (`RevokeAllSessions`).
3. Set `users.status = 'deleted'` (soft delete, not `DELETE` row).
4. Return `{ deleted: true }`.
5. Subsequent logins for this user return `ErrInvalidCredentials` because `Login` checks `status == 'active'`.

> **Hard delete is blocked** by `devices.owner_id REFERENCES users(id) ON DELETE RESTRICT`. A user with owned devices cannot be hard-deleted until they transfer or decommission all devices.

## 6. Role-Based Access Control (RBAC)

### 6.1 Global Account Role

Every user has a single `role` stored in `users.role`. It is embedded in the JWT claims as `"role"`:

```
owner   > admin   > member   > viewer
```

| Role | Intended Use |
|---|---|
| `owner` | Default for all self-registered users. Full platform access. |
| `admin` | Platform-level admin (future: dashboard, user management). |
| `member` | Household member — standard device access without ownership controls. |
| `viewer` | Read-only access. Cannot issue commands or change settings. |

**Go middleware to gate routes by role:**

```go
// internal/middleware/auth.go
func RequireRole(roles ...string) gin.HandlerFunc

// Usage in router:
adminGroup := root.Group("/admin", requireAuth, middleware.RequireRole("admin", "owner"))
```

**⚠ Current state:** `RequireRole` is implemented but not yet applied to any route. All `requireAuth`-protected routes accept all roles. Add `RequireRole` to routes as engine-specific policies are defined.

### 6.2 Device-Level Permission Checks

The global `role` field intentionally does **not** control device access. Device permission is separate:

```
Device Access Level    Check
─────────────────────────────────────────────────────────
owner                  device.owner_id == userID
admin                  device_admins WHERE device_id = X AND user_id = Y
(no access)            neither of the above
```

This maps to two Gin middleware functions in `devices/handlers.go`:

```go
// access() — owner OR admin may proceed
func (h *Handler) access() gin.HandlerFunc

// ownerOnly() — only owner may proceed
func (h *Handler) ownerOnly() gin.HandlerFunc
```

```
Route                                       Guard
────────────────────────────────────────────────────────────────────────
GET  /devices/:id                           access()      owner OR admin
PATCH /devices/:id                          access()      owner OR admin
DELETE /devices/:id                         ownerOnly()   owner only
POST /devices/:id/transfer-ownership        ownerOnly()   owner only
POST /devices/:id/admins                    ownerOnly()   owner only
DELETE /devices/:id/admins/:userId          ownerOnly()   owner only
GET  /devices/:id/history                   access()      owner OR admin
POST /devices/:id/mqtt-credentials          access()      owner OR admin
DELETE /devices/:id/mqtt-credentials        access()      owner OR admin
GET  /devices/:id/mqtt-topics               access()      owner OR admin
```

### 6.3 Planned Fine-Grained Permissions (`permissions` table)

The schema includes a `permissions` table for per-device, per-user, per-role permissions with revocation timestamps:

```sql
CREATE TABLE permissions (
    id          UUID PRIMARY KEY,
    device_id   UUID REFERENCES devices(id) ON DELETE CASCADE,
    user_id     UUID REFERENCES users(id) ON DELETE CASCADE,
    role        TEXT CHECK (role IN ('owner','admin','viewer')),
    granted_by  UUID REFERENCES users(id) ON DELETE SET NULL,
    granted_at  TIMESTAMPTZ,
    revoked_at  TIMESTAMPTZ,     -- NULL = currently active
    UNIQUE (device_id, user_id)
);
```

**Current status:** Schema-only. No engine reads or writes this table yet. It is designed to replace/augment `device_admins` when a `viewer` role on a device is needed (read-only smart home guests), or when time-bounded access grants are required.

**Implementation path:**
1. Create `internal/engines/permissions` package.
2. Migrate `CanAccess()` to check `permissions` table in addition to (or instead of) `device_admins`.
3. Update Invitations engine to write a `permissions` row on acceptance (see §9).

## 7. Device Ownership & Permission Model

### 7.1 Ownership Concept

Every `devices` row has exactly one `owner_id`. The owner is the account that registered the device.

```
┌─────────────────────────────────────────────────────────────┐
│  Device  (e.g. smart lamp)                                  │
│                                                             │
│  owner_id ──► User A  (full control, can transfer, delete)  │
│                                                             │
│  device_admins ──► User B  (can read, update, send commands)│
│               ──► User C  (same)                            │
│                                                             │
│  (future) permissions.viewer ──► User D  (read-only)        │
└─────────────────────────────────────────────────────────────┘
```

### 7.2 Permission Matrix

| Action | Owner | Admin | Future: Viewer | No Access |
|---|:---:|:---:|:---:|:---:|
| List device | ✅ | ✅ | ✅ | ❌ |
| Get device detail | ✅ | ✅ | ✅ | ❌ |
| Update device name/firmware | ✅ | ✅ | ❌ | ❌ |
| View device history | ✅ | ✅ | ✅ | ❌ |
| Issue MQTT credentials | ✅ | ✅ | ❌ | ❌ |
| Grant admin to another user | ✅ | ❌ | ❌ | ❌ |
| Revoke admin from a user | ✅ | ❌ | ❌ | ❌ |
| Transfer ownership | ✅ | ❌ | ❌ | ❌ |
| Delete / decommission device | ✅ | ❌ | ❌ | ❌ |

### 7.3 `device_admins` Table

```sql
CREATE TABLE device_admins (
    device_id  UUID REFERENCES devices(id) ON DELETE CASCADE,
    user_id    UUID REFERENCES users(id) ON DELETE CASCADE,
    granted_by UUID REFERENCES users(id) ON DELETE SET NULL,
    granted_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    PRIMARY KEY (device_id, user_id)
);
```

There is no `revoked_at` column — admin access is binary. Revocation is a hard `DELETE` of the row (`RemoveAdmin`). The history for who was granted/revoked lives in `device_history`.

### 7.4 Device History

Every ownership-modifying action appends an immutable row to `device_history`:

```sql
CREATE TABLE device_history (
    event_type TEXT CHECK (event_type IN (
        'registered', 'updated', 'ownership_transferred',
        'removed', 'admin_granted', 'admin_revoked'
    )),
    actor_user_id UUID REFERENCES users(id) ON DELETE SET NULL,
    metadata      JSONB,   -- context-specific payload
    created_at    TIMESTAMPTZ
);
```

This is separate from the global `audit_logs` table — device history is user-facing ("who did what to my device"), audit logs are ops-facing ("all backend events by resource type").

## 8. Device Management Service — Auth Integration

### 8.1 Device Registration

```
POST /cloud/devices
Authorization: Bearer <access_token>
{
  "name": "Living Room Lamp",
  "deviceType": "smart_lamp",
  "microcontrollerId": "ESP32-ABC123",
  "macAddress": "AA:BB:CC:DD:EE:FF",
  "firmwareVersion": "1.2.0",
  "capabilities": ["onoff", "brightness", "color_temp"]
}
```

- The authenticated user becomes `owner_id`.
- `mac_address` is the deduplication key — registering the same MAC returns `409 DEVICE_ALREADY_REGISTERED`.
- Status is set to `pending` until the device connects via MQTT and the backend marks it `online`.

### 8.2 Accessing the Device Service from Auth Context

The Devices Engine needs to resolve an email address to a `user_id` when granting admin access or transferring ownership (the UI sends an email, not a UUID). This is done through a narrow interface to avoid circular imports:

```go
// devices/service.go
type UserLookup interface {
    FindUserIDByEmail(email string) (uuid.UUID, error)
}

// auth/service.go implements UserLookup:
func (s *Service) FindUserIDByEmail(email string) (uuid.UUID, error)
```

The Auth Service is wired as the `UserLookup` implementation at startup in `cmd/api/main.go`.

### 8.3 Device List — Owner + Admin Combined Query

```go
// devices/repository.go — ListForUser
base := r.db.Model(&models.Device{}).
    Where("owner_id = ? OR id IN (SELECT device_id FROM device_admins WHERE user_id = ?)",
        userID, userID)
```

One query returns all devices the user can access (owned + administered). Pagination (`page`, `perPage`, max 100) is applied after the combined count.

### 8.4 Admin Grant & Revoke

```
POST /cloud/devices/:deviceId/admins
Authorization: Bearer <access_token>  (owner only)
{ "userEmail": "friend@example.com" }

DELETE /cloud/devices/:deviceId/admins/:userId
Authorization: Bearer <access_token>  (owner only)
```

Both operations write an event to `device_history` and `audit_logs`. The `DeviceDTO` returned from `Get` includes `adminUserIds []string` so the mobile client can render the current admin list.

## 9. Invitations Engine

### 9.1 Overview

The Invitations Engine is the **owner-driven push model** for sharing device access: the owner proactively sends an invitation to a user (by email or username); the invitee accepts or declines.

### 9.2 Domain Model

```typescript
// Mobile type (services/cloud-api.ts)
export interface CloudInvitation {
  id: string;
  fromUserId: string;
  fromUserName: string;
  toEmail?: string;       // target by email
  toUsername?: string;    // OR by username
  deviceId: string;
  deviceName: string;
  permissions: string[];  // e.g. ["view", "control", "admin"]
  expiresAt: string;
  status: "pending" | "accepted" | "declined" | "expired";
  message?: string;
  createdAt: string;
}
```

### 9.3 Proposed Database Schema

```sql
CREATE TABLE invitations (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    from_user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    to_email     TEXT,           -- exactly one of to_email or to_user_id
    to_user_id   UUID REFERENCES users(id) ON DELETE CASCADE,
    device_id    UUID NOT NULL REFERENCES devices(id) ON DELETE CASCADE,
    permissions  TEXT[] NOT NULL DEFAULT '{}',
    message      TEXT,
    status       TEXT NOT NULL DEFAULT 'pending'
                 CHECK (status IN ('pending','accepted','declined','expired','cancelled')),
    expires_at   TIMESTAMPTZ NOT NULL DEFAULT now() + INTERVAL '7 days',
    responded_at TIMESTAMPTZ,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_invitations_to_user   ON invitations (to_user_id, status);
CREATE INDEX idx_invitations_from_user ON invitations (from_user_id, status);
CREATE INDEX idx_invitations_device    ON invitations (device_id);
```

### 9.5 Accept Flow — Connection to Device Permissions

Accepting an invitation must grant the invitee device access. Implementation has two options:

**Option A — Write to `device_admins` (current RBAC table, binary admin):**
```sql
INSERT INTO device_admins (device_id, user_id, granted_by)
VALUES ($deviceId, $inviteeId, $senderId)
ON CONFLICT DO NOTHING;
UPDATE invitations SET status='accepted', responded_at=now() WHERE id=$id;
```

**Option B — Write to `permissions` (fine-grained, recommended for future):**
```sql
INSERT INTO permissions (device_id, user_id, role, granted_by)
VALUES ($deviceId, $inviteeId, $role, $senderId)
ON CONFLICT (device_id, user_id) DO UPDATE SET role=$role, revoked_at=NULL;
UPDATE invitations SET status='accepted', responded_at=now() WHERE id=$id;
```

Option B requires `CanAccess()` to be updated to also check the `permissions` table.

### 9.6 Expiry Handling

A background job (or request-time check) should mark invitations as `expired` when `expires_at < now()`. Accepting an expired invitation returns `400 INVITATION_EXPIRED`. This can be implemented as a Gin middleware on the accept endpoint:

```go
if invitation.ExpiresAt.Before(time.Now()) {
    return ErrInvitationExpired
}
```

## 10. Access Requests Engine

### 10.1 Overview

The Access Requests Engine is the **guest-driven pull model** for sharing: any user can request access to a device owned by another user; the owner approves or rejects.

### 10.2 Domain Model

```typescript
// Mobile type (services/cloud-api.ts)
export interface CloudAccessRequest {
  id: string;
  requesterId: string;
  requesterName: string;
  requesterEmail: string;
  deviceId?: string;
  deviceName?: string;
  roomId?: string;           // future: room-level requests
  mcDeviceId?: string;       // specific microcontroller
  lampId?: string;           // specific lamp within a device
  permissionLevel: string;   // "viewer" | "admin"
  message?: string;
  status: "pending" | "approved" | "rejected" | "blocked";
  requestedAt: string;
  respondedAt?: string;
}
```

### 10.3 Proposed Database Schema

```sql
CREATE TABLE access_requests (
    id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    requester_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    target_owner_id  UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    device_id        UUID REFERENCES devices(id) ON DELETE CASCADE,
    permission_level TEXT NOT NULL DEFAULT 'viewer'
                     CHECK (permission_level IN ('viewer','admin')),
    message          TEXT,
    status           TEXT NOT NULL DEFAULT 'pending'
                     CHECK (status IN ('pending','approved','rejected','blocked')),
    responded_by     UUID REFERENCES users(id) ON DELETE SET NULL,
    responded_at     TIMESTAMPTZ,
    created_at       TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_access_requests_owner     ON access_requests (target_owner_id, status);
CREATE INDEX idx_access_requests_requester ON access_requests (requester_id, status);
CREATE INDEX idx_access_requests_device    ON access_requests (device_id);
```

**`status = 'blocked'`** means the owner has blocked this requester for this device — re-submitting a request from the same user for the same device will be rejected immediately without notifying the owner.

### 10.4 API Endpoints (to implement)

| Method | Path | Auth | Guard | Description |
|---|---|---|---|---|
| `POST` | `/api/engines/access-requests` | Required | Any authenticated user | Submit a request |
| `GET` | `/api/engines/access-requests` | Required | — | List own requests (sent + incoming) |
| `POST` | `/api/engines/access-requests/:id/approve` | Required | Device owner only | Approve + grant access |
| `POST` | `/api/engines/access-requests/:id/reject` | Required | Device owner only | Reject |
| `POST` | `/api/engines/access-requests/:id/block` | Required | Device owner only | Reject + block requester |

### 10.5 Approve Flow

```
POST /api/engines/access-requests/:id/approve
{ "permissions": ["viewer"] }
```

1. Verify caller is the device owner (`device.owner_id == userID`).
2. Verify request status is `pending`.
3. Grant access via `device_admins` or `permissions` table based on `permissionLevel`.
4. Set `status = 'approved'`, `responded_by`, `responded_at`.
5. Notify requester (Notifications Engine).

### 10.6 Identification — How a Requester Finds a Target Device

A requester does not necessarily know the device UUID. Three supported targeting modes:

| Mode | Fields | Resolution |
|---|---|---|
| Direct device | `deviceId` | Direct FK lookup |
| By owner email | `targetEmail` | Resolve to `target_owner_id` → list their devices |
| By owner username | `targetUsername` | Resolve to `target_owner_id` → list their devices |

The resolution from email/username to owner ID uses the same `UserLookup` interface the Devices Engine uses.

## 11. Ownership Transfer

Ownership transfer is **fully implemented** in the Devices Engine.

### 11.1 Flow

```
POST /cloud/devices/:deviceId/transfer-ownership
Authorization: Bearer <access_token>  (owner only — ownerOnly() guard)
{ "newOwnerEmail": "newowner@example.com" }
```

1. Verify caller is current owner (`ownerOnly()` guard pre-checks this).
2. Resolve `newOwnerEmail` → `newOwnerID` via `UserLookup.FindUserIDByEmail`.
3. `UPDATE devices SET owner_id = newOwnerID`.
4. **Auto-demote:** `INSERT INTO device_admins (device_id, user_id=currentOwner, granted_by=newOwner)` — previous owner retains admin access automatically.
5. Append `ownership_transferred` event to `device_history`.
6. Audit log.
7. Return updated `DeviceDTO`.

### 11.2 Mobile UI

The mobile app exposes this through `/ownership-transfer` screen, which:
- Shows a device picker.
- Accepts target email.
- Has a "Keep admin access" toggle (currently hardcoded to true in the backend).

> **⚠ Current gap:** The mobile API call sends `{ toEmail, previousOwnerBecomesAdmin }` but the backend `TransferOwnershipRequest` only reads `newOwnerEmail`. The `previousOwnerBecomesAdmin` flag is ignored — the backend always adds the previous owner as admin. This should be wired up: read the flag and skip the `AddAdmin` call if `previousOwnerBecomesAdmin = false`.

## 12. MQTT Device Identity & Authorization

### 12.1 Schema

```sql
CREATE TABLE mqtt_device_identities (
    id              UUID PRIMARY KEY,
    device_id       UUID REFERENCES devices(id) ON DELETE CASCADE,
    mqtt_client_id  TEXT NOT NULL UNIQUE,
    mqtt_username   TEXT NOT NULL,
    credential_hash TEXT NOT NULL,   -- hashed MQTT password
    issued_at       TIMESTAMPTZ,
    expires_at      TIMESTAMPTZ,
    revoked_at      TIMESTAMPTZ      -- NULL = active
);
```

### 12.2 MQTT Credential Endpoints

All three endpoints sit on the devices group and are protected by `devicesAccessMiddleware` (owner OR admin check):

| Method | Path | Description |
|---|---|---|
| `POST` | `/cloud/devices/:deviceId/mqtt-credentials` | Issue new MQTT credential set |
| `DELETE` | `/cloud/devices/:deviceId/mqtt-credentials` | Revoke active credentials |
| `GET` | `/cloud/devices/:deviceId/mqtt-topics` | Get scoped MQTT topic list |

### 12.3 Authorization Model

The backend **never** gives a mobile client the broker's own admin credentials. Instead it issues scoped `mqtt_device_identities` that the MQTT broker's authentication plugin (e.g. EMQX, Mosquitto auth plugin) verifies:

```
Mobile App → POST /devices/:id/mqtt-credentials
           ← { clientId, username, password }
                         │
                         ▼
                Mobile connects to MQTT broker
                broker calls backend auth plugin
                plugin verifies username+hash
                topics scoped to device/ prefix
```

This means revoking a credential at the backend (`DELETE /mqtt-credentials`) prevents the device or any app holding old credentials from re-connecting after their current TCP session drops.

## 13. Security Hardening Properties

| Property | Implementation |
|---|---|
| **Password storage** | bcrypt with default cost (10 rounds). No plain text, no MD5/SHA1. |
| **Token rotation** | Refresh tokens are single-use. Old session blacklisted on each rotation. |
| **Session blacklist** | Redis (prod) / in-memory (dev). Allows immediate logout without waiting for JWT TTL. |
| **Email enumeration protection** | `POST /auth/request-password-reset` always returns 200. |
| **Status gate** | `Login` rejects `status != 'active'` — suspended or deleted accounts cannot authenticate. |
| **Rate limiting** | Token-bucket rate limiter on all routes under `/cloud`. Default: configurable RPM + burst. Uses Cache backend (Redis in prod). |
| **CORS** | Configured per environment via `config.CORSOrigins`. Reject unknown origins. |
| **Request IDs** | Every request tagged with a UUID in `X-Request-ID` for traceability. |
| **Structured logging** | Every request logged with `requestId, method, path, status, durationMs, userId`. |
| **Panic recovery** | `middleware.Recovery` catches panics and returns `500` without crashing the server. |
| **MQTT scope isolation** | Devices get credential-scoped MQTT identities — they cannot publish/subscribe to other devices' topics. |
| **Soft delete only** | Deleted users are marked `status='deleted'`, never hard-deleted, preserving audit and device history FK integrity. |

### 16.1 Auth State Machine (CloudAuthContext)

```
┌─────────────────────────────────────────────────────────┐
│  App Start                                              │
│       │                                                 │
│       ▼                                                 │
│  isLoading = true                                       │
│  Restore: getStoredUser() + getStoredUsername()         │
│         + getCachedSyncData()                           │
│       │                                                 │
│       ├─ user found ──► isAuthenticated = true          │
│       │                  isLoading = false              │
│       │                                                 │
│       └─ no user ────► isAuthenticated = false          │
│                          isLoading = false              │
│                          → AuthGuard → /login           │
└─────────────────────────────────────────────────────────┘

Login / Register
       │
       ▼
  CloudAPI.login/register()
       │ success
       ▼
  storeAuth(auth) + storeUsername(username)
  applyUser(user)  → setUser, AsyncStorage update
       │
       ▼ (non-blocking)
  syncAllData() → setSyncData → bumpSyncKey
```

### 16.2 AuthGuard

const PUBLIC_ROUTES = new Set(["login","forgot-password","verify-email","+not-found"]);

|---|---|---|---|

### 16.4 Sync Key Pattern

`syncKey` is an integer in `CloudAuthContext` that increments after every successful `syncAllData()`. `LumaContext` watches it to know when to re-read cloud devices without needing prop-drilling or event buses:

## 18. Extension Guide

### 18.1 Adding the Invitations Engine

1. **Migration:** Add `000006_invitations.up.sql` (schema in §9.3).
2. **Package:** Create `internal/engines/invitations/` with `dto.go`, `repository.go`, `service.go`, `handlers.go`.
3. **Wire:** In `cmd/api/main.go`, instantiate `invitations.NewService(repo, devicesRepo, usersLookup)` and register routes on `root.Group("/api/engines/invitations", requireAuth)`.
4. **Accept flow:** On accept, call either `devicesRepo.AddAdmin()` or write to `permissions` table depending on `permissions` field in the invitation.
5. **Notification:** After accept/decline/cancel, call `notificationsService.Create(userID, type, title, body)`.

### 18.2 Adding the Access Requests Engine

Same pattern as §18.1 but uses `000007_access_requests.up.sql`. The `approve` handler must call the Devices Service `GrantAdmin` rather than writing to `device_admins` directly — this preserves the audit trail.

### 18.3 Activating Fine-Grained `viewer` Role

1. Update `CanAccess(deviceID, userID)` to also `SELECT FROM permissions WHERE device_id=$1 AND user_id=$2 AND revoked_at IS NULL`.
2. Add a third middleware `viewerAccess()` that checks `CanAccess` but only allows read (`GET`) methods.
3. Apply `viewerAccess()` to `GET /devices/:id`, `GET /devices/:id/history`.
4. Update `TransferOwnership` and `GrantAdmin` to write to `permissions` instead of (or in addition to) `device_admins`.

### 18.4 Applying `RequireRole` to Admin Routes

To protect a future admin dashboard:

```go
// In router.go
adminGroup := root.Group("/admin", requireAuth, middleware.RequireRole("admin", "owner"))
adminGroup.GET("/users", adminHandler.ListUsers)
adminGroup.PATCH("/users/:id/suspend", adminHandler.SuspendUser)
```

### 18.5 Wiring Email Delivery

Password reset and email verification tokens are generated and stored but delivery is currently only logged. To wire a real email provider:

1. Add an `EmailSender` interface to `internal/services/email/`.
2. Inject it into `auth.Service`.
3. Call `emailSender.Send(to, subject, body)` in `RequestPasswordReset` and `RequestEmailVerification`.
4. In Replit environment, use the environment-secrets skill to store SMTP credentials as `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS`.

### 18.6 Redis for Production Session Blacklist

The session blacklist defaults to in-memory, which is lost on restart. To use Redis:

1. Set `REDIS_URL` environment secret.
2. The backend auto-detects `REDIS_URL` at startup and switches the `Cache` backend to Redis.
3. The `Blacklist` struct in `internal/engines/auth/blacklist.go` uses the `Cache` interface — no code changes needed.
