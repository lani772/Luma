├── hooks/                  # useColors, useInstallPrompt, useEngines

| Layer | Implementation | Screens |

**Gap:** There is no authentication screen. The app starts directly on the dashboard. The cloud backend has a full JWT auth system that is never reachable from the mobile app.

### 2.3 State Management

```
LumaContext (React useState)
  ├── lamps[]              ← in-memory, from luma-data.ts
  ├── scenes[]             ← in-memory, from luma-data.ts
  ├── users[] + lumaUsers[]← two parallel user models (see §5.1)
  ├── notifications[]      ← in-memory only, not persisted
  ├── microcontrollers[]   ← in-memory only, not in DB schema
  ├── mcDevices[]          ← in-memory only
  ├── mcUsers[]            ← in-memory only
  └── lampAutomations{}    ← in-memory only
```

**Critical gap:** `LumaContext` never makes a single HTTP call. Every mutation is local state only. There is no API layer wired from the mobile app to either the `api-server` or `luma-cloud-backend`.

### 2.4 CoreEngine (12 Sub-Engines)

The CoreEngine is a well-designed singleton with waveset boot ordering:

| Wave | Engines |
|---|---|
| 0 — Foundation | DatabaseEngine, EventEngine |
| 1 — Security | SecurityEngine, ExtensionEngine |
| 2 — Comms/Discovery | PermissionEngine, MQTTCommunicationEngine, DiscoveryEngine |
| 3 — Device layer | DeviceManagementEngine |
| 4 — Features | NotificationEngine, AutomationEngine, FirmwareEngine, DashboardEngine |

**Gap:** `CoreEngine.boot()` is **never called** in `app/_layout.tsx`. The CoreEngine singleton is imported but never started. All 12 engines are instantiated but stay in `idle` state for the lifetime of the app.

## 3. Backend Architecture Assessment

### 3.1 api-server (Node.js / Express 5 / TypeScript)

**Role:** Local hardware gateway — communicates with ESP32 devices over USB, WiFi, and MQTT on the home LAN.

**Architecture:**

```
InternalAPIGateway
  ├── MessageBus (Node EventEmitter)
  └── 6 Engines (all extend BaseEngine)
       ├── firmware_engine      — OTA management
       ├── device_engine        — device registry (in-memory)
       ├── wifi_engine          — hotspot/WiFi management
       ├── mqtt_engine          — local MQTT broker bridge
       ├── usb_engine           — serial/USB provisioning
       └── firmware_upload_engine — binary upload handler
```

**Exposed Routes:**
```
GET  /api/engines                        → list engines + status
POST /api/engines/:engineId/command      → send command to engine
GET  /api/engines/devices/all            → device registry snapshot
GET  /health                             → liveness probe
```

**Strengths:**
- Clean engine abstraction with `BaseEngine` interface
- MessageBus decouples engines from each other
- Pino structured logging

**Weaknesses / Gaps:**
- No persistence: `device_engine` holds state in-memory; all state lost on restart
- No authentication: any process on the LAN can issue commands
- No MQTT topic standard: the engine publishes on topics but there is no ACL
- No WebSocket: the mobile app cannot subscribe to live device events via the api-server
- No schedule execution: schedules stored in LumaContext but never executed server-side
- Only 3 real routes — not yet a device management API

---

### 3.2 luma-cloud-backend (Go / Gin)

**Role:** Cloud authority — authentication, ownership, permissions, MQTT credential issuance, firmware delivery, sync, backup, audit, analytics.

**Architecture:**

```
Gin Router (/cloud prefix)
  ├── Middleware: Recovery, RequestID, StructuredLogging, CORS, RateLimit
  ├── RequireAuth (JWT validation + TokenBlacklist)
  └── Engines:
       ├── auth        — register, login, refresh, logout, password reset
       ├── users       — profile, multi-device (user_phones)
       ├── devices     — register, ownership, admin grants, history
       ├── mqtt        — scoped credential issuance (EMQX adapter)
       ├── firmware    — release management, channels (stable/beta)
       ├── deployment  — OTA campaign deployment
       ├── notifications — push notifications (FCM/APNs)
       ├── sync        — per-phone resource version sync
       └── backup      — cloud backup/restore
```

**Database Schema — Implemented Tables (Phase 1):**

| Table | Purpose |
|---|---|
| `users` | UUID PK, email (citext), bcrypt hash, role, subscription tier |
| `user_phones` | Multi-device per user, FCM/APNs push tokens |
| `sessions` | Opaque refresh tokens (hashed), per-phone |
| `password_reset_tokens` | Time-limited reset tokens |
| `email_verification_tokens` | Email verification flow |
| `devices` | Owner FK, MAC address (unique), firmware version, capabilities JSONB |
| `device_admins` | M2M — devices shared to non-owner users |
| `device_history` | Audit events for device lifecycle |
| `mqtt_device_identities` | Scoped MQTT credentials per device |

**Database Schema — Reserved Tables (Phase 2, schema exists, no engine yet):**

| Table | Purpose |
|---|---|
| `firmware_releases` | Binary metadata, checksum SHA-256, rollback flag |
| `firmware_downloads` | Download audit per device |
| `permissions` | Fine-grained device permissions (owner/admin/viewer) |
| `schedules` | Device schedules with `time_config: JSONB` and `action: JSONB` |
| `scenes` | User-owned scenes with `actions: JSONB` |
| `notifications` | Persistent per-user notifications |
| `audit_logs` | Full system audit trail |
| `analytics_events` | Raw telemetry events |
| `analytics_daily_rollups` | Pre-aggregated daily metrics |
| `sync_states` | Per-phone resource version tracking |
| `sync_history` | Conflict-aware sync log |
| `backups` | Backup metadata (object storage path) |

**Strengths:**
- Clean engine-per-domain architecture with handler/service/repository separation
- Stable machine-readable error codes (`TOKEN_EXPIRED`, `DEVICE_ALREADY_REGISTERED`, etc.)
- JWT + opaque refresh token rotation with Redis blacklist
- RPM-based rate limiting per IP
- Structured logging with `log/slog`
- MQTT adapter interface (`pkg/mqttadapter`) is broker-agnostic (Paho + noop implementations)
- Full audit design already in schema

**Weaknesses / Gaps:**
- **No microcontroller table** in any migration
- **`role` column on `users`** has only 4 values (`owner/admin/member/viewer`) — inconsistent with the frontend's `LumaRole` (`owner/admin/member/guest`)
- Phase 2 tables exist but none of their engines are wired: schedules, scenes, automations, rooms have schema but no REST routes
- No room management table or engine
- No automation table or engine
- No WebSocket endpoint for real-time device event push to mobile
- The `permissions` table overlaps with `device_admins` — redundant access control design
- No microcontroller concept: backend models ESP32s as flat `devices` with `microcontroller_id TEXT` field, but the mobile app treats MCUs as first-class entities with their own lifecycle

## 4. MQTT Broker Integration Assessment

### 4.1 Current State

The system has **two separate MQTT topic naming conventions** that are incompatible:

**Frontend (`src/modules/mqtt/MQTTTopics.ts`):**
```
device/{deviceId}/status
device/{deviceId}/command
device/{deviceId}/response
device/{deviceId}/event
device/{deviceId}/firmware
device/{deviceId}/schedule
device/{deviceId}/permission
device/{deviceId}/heartbeat
device/{deviceId}/logs
device/{deviceId}/sync
```

**Cloud backend (`internal/engines/mqtt/topics.go`):**
```
luma/devices/{deviceId}/state
luma/devices/{deviceId}/telemetry
luma/devices/{deviceId}/commands
luma/devices/{deviceId}/status
```

**This is a blocking inconsistency.** The mobile app publishes to `device/{id}/command` but the ESP32 device firmware would be written to subscribe to `luma/devices/{id}/commands`. Commands will never be received.

### 4.2 Connection Flow (Current)

```
Mobile App
  ├── MQTTManager.connectAll({ cloud: { host, port, username, password } })
  │    ├── MQTTConnection("cloud") → tries native NativeModules.MqttClient
  │    │    └── Falls back to simulated mqtt-client-engine (Expo Go)
  │    └── MQTTConnection("local") → same pattern
  └── Publishes to device/{id}/command   ← WRONG TOPIC NAMESPACE
```

**The cloud backend issues MQTT credentials** via `POST /cloud/devices/{deviceId}/mqtt-credentials` — but the mobile app never calls this endpoint. The app hardcodes no broker credentials at all.

### 4.3 MQTT Architecture Gaps

| Concern | Current State | Required |
|---|---|---|
| Topic namespace | Two conflicting schemes | One canonical scheme |
| Credential issuance | Backend can issue; app never requests | App must call `/cloud/devices/{id}/mqtt-credentials` on login |
| QoS | Not defined | QoS 1 for commands, QoS 0 for telemetry |
| Retained messages | Not configured | `/state` and `/status` topics should be retained |
| Last Will & Testament | Not implemented | Each device should set LWT on `/status` → `{"online":false}` |
| Topic ACLs | EMQX adapter exists but no ACL rules published | Broker must enforce per-device scope |
| Shared subscriptions | Not implemented | Useful for multi-phone households |
| Reconnect strategy | `MQTTConnection` has exponential backoff | Good — keep as-is |
| Offline queue | `MQTTQueue` module exists | Good — keep, expose size synchronously |

## 5. Critical Inconsistencies & Gaps

### 5.1 Two User Models

The frontend has two parallel user type systems that are never reconciled:

**Model A — `User` (legacy):**
```typescript
{ id: number; name: string; email: string;
  role: "Admin" | "Manager" | "Operator" | "Viewer";
  status: "active" | "inactive" }
```

**Model B — `LumaUser` (RBAC):**
```typescript
{ id: number; name: string; email: string;
  role: "owner" | "admin" | "member" | "guest";
  lampIds: string[];
  perms: Record<string, boolean> }
```

**Cloud backend `users` table:**
```sql
role TEXT CHECK (role IN ('owner', 'admin', 'member', 'viewer'))
```

All three are different. `Model A` is used in `app/(tabs)/users.tsx`. `Model B` is used in most other screens. The cloud backend uses a fourth variant (`viewer` instead of `guest`).

**Resolution needed:** Canonicalize on a single role enum. Recommended: `owner | admin | member | guest` (Model B / mobile RBAC). Remove `Model A` entirely. Update cloud backend to `viewer → guest`.

### 5.2 No Microcontroller in the Database

The frontend treats `Microcontroller` as a first-class entity with:
- `id`, `name`, `model`, `firmware`, `mac`, `ip`, `status`
- `configVersion`, `hardwareVersion`, `lastSync`
- Child `MCDevice[]` with full CRUD

The cloud backend's `devices` table has `microcontroller_id TEXT NOT NULL` — treating the MCU merely as a metadata string on a device. There is no `microcontrollers` table.

This means: MCU registration, ownership, firmware management per MCU, MCU-level permissions — all unimplementable until the schema is extended.

### 5.3 CoreEngine Never Booted

```typescript
// app/_layout.tsx — CoreEngine is NOT called
export default function RootLayout() {
  // ... fonts, splash screen
  return (
    <SafeAreaProvider>
      <LumaProvider>
        {/* CoreEngine.boot() is never invoked */}
      </LumaProvider>
    </SafeAreaProvider>
  );
}
```

The `CoreEngine` singleton is imported in `engines/core/index.ts` and re-exported, but `CoreEngine.boot()` is never called. All 12 engines remain in `idle` state. The `MQTTCommunicationEngine`, `SecurityEngine`, `PermissionEngine`, `AutomationEngine` — all idle.

### 5.4 Two MQTT Layers (Fragmented)

There are two separate MQTT architecture layers:

1. `engines/core/MQTTCommunicationEngine.ts` — part of CoreEngine, never started
2. `src/modules/mqtt/MQTTManager.ts` + `MQTTContext.tsx` — wired to UI, simulated

These overlap significantly. The `MQTTCommunicationEngine` wraps `MQTTManager` conceptually but neither calls the other. This needs to be unified: CoreEngine should own and start MQTTManager.

### 5.5 Automation Model Is Too Flat

Current frontend `AutomationRule`:
```typescript
{ id: string; name: string; trigger: string; action: string; enabled: boolean; priority: number }
```

`trigger` and `action` are free-form strings ("Sunset", "Motion Detected"). No structured conditions, no nested logic, no time windows, no multi-trigger support.

The database schema has no `automations` table at all.

### 5.6 Device Type System Is "Lamps Only"

The entire device model is typed as `Lamp`. Features specific to lamps (brightness, colorTemp, rgb, power monitoring) are in every device. There's no polymorphic device model for:
- Fans (speed, oscillation)
- Sensors (temperature, humidity, motion, door/window)
- Switches (momentary, latching)
- Locks (lock/unlock, PIN)
- Outlets (power monitoring)

## 6. Device & Microcontroller Management Workflows

### 6.1 Microcontroller Full Lifecycle

```
┌─────────────────────────────────────────────────────────────┐
│                  Microcontroller Lifecycle                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  UNREGISTERED                                               │
│      │                                                      │
│      ▼ Physical pairing (USB / BLE / WiFi AP)               │
│  PROVISIONING                                               │
│      │ WiFi credentials + broker URL pushed                 │
│      ▼                                                      │
│  PENDING    ← cloud creates record, status = "pending"      │
│      │ MCU connects to broker, sends heartbeat              │
│      ▼                                                      │
│  ACTIVE     ← status = "online"                             │
│      │                                                      │
│      ├──── Rename → updateMicrocontroller()                 │
│      ├──── Add Devices → addMCDevice()                      │
│      ├──── Firmware Update → firmwareEngine.push()          │
│      ├──── Backup Config → backupEngine.snapshot()          │
│      ├──── Transfer Ownership → devicesService.transfer()   │
│      ├──── Suspend → status = "suspended"                   │
│      └──── Delete → cascade-delete all child devices        │
│                                                             │
│  OFFLINE    ← LWT fires, status = "offline"                 │
│      │                                                      │
│      └──── Recovery → reconnect → ACTIVE                    │
│                                                             │
│  DECOMMISSIONED ← permanent removal                         │
└─────────────────────────────────────────────────────────────┘
```

**Required DB change:** Add `microcontrollers` table to cloud backend.

```sql
CREATE TABLE microcontrollers (
    id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    owner_id         UUID NOT NULL REFERENCES users(id) ON DELETE RESTRICT,
    name             TEXT NOT NULL,
    model            TEXT NOT NULL,
    mac_address      MACADDR NOT NULL UNIQUE,
    ip_address       INET,
    firmware_version TEXT,
    hardware_version TEXT,
    config_version   INTEGER NOT NULL DEFAULT 0,
    status           TEXT NOT NULL DEFAULT 'pending'
                     CHECK (status IN ('pending','online','offline','suspended','decommissioned')),
    registered_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    last_seen_at     TIMESTAMPTZ,
    last_sync_at     TIMESTAMPTZ,
    created_at       TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at       TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Update devices table to use UUID FK instead of TEXT
ALTER TABLE devices
    ADD COLUMN microcontroller_id_fk UUID REFERENCES microcontrollers(id) ON DELETE CASCADE;
```

### 6.2 Device Registration Workflow

```
Mobile App                     Cloud API                     MCU / ESP32
───────────────────────────────────────────────────────────────────────
  1. User taps "Add Device"
  2. WiFi provisioning via
     USB or BLE AP mode         ──── POST /cloud/microcontrollers ────►
                                 ◄─── { id, mqttCredentials } ────────
  3. App pushes MQTT creds
     + broker URL to MCU via
     USB/BLE provisioning channel
                                                                ◄── MCU reboots
                                                                MCU connects to broker
                                                                MCU publishes heartbeat
  4. MCU heartbeat received
     by broker → webhook        ◄─── webhook: device online ──────────
  5. status → "online"          ──── PATCH /cloud/devices/{id} ───────►
  6. UI updated via sync
```

### 6.3 Ownership Transfer

```
Current Owner → POST /cloud/devices/{id}/transfer → { newOwnerEmail }
  ├── Validate newOwner exists
  ├── Validate caller is current owner
  ├── Revoke all device_admins for old owner
  ├── Update devices.owner_id
  ├── Insert device_history record
  ├── Revoke old owner's MQTT credentials for this device
  ├── Issue new MQTT credentials for new owner
  └── Push notification to both parties
```

## 7. Device Action Workflows

### 7.1 Universal Action Pipeline

Every device action — regardless of type (toggle, dim, lock, fan speed) — must flow through this pipeline:

```
User Gesture
    │
    ▼
[1] Permission Check
    PermissionEngine.check(userId, deviceId, action)
    └── Blocked? → show error toast, audit log, return
    │
    ▼
[2] Optimistic UI Update
    LumaContext.updateLamp(id, { on: true })
    └── Immediate UI feedback (no waiting for broker)
    │
    ▼
[3] Command Signing (if deviceKey available)
    SecurityEngine.signCommand({ device, action, params, nonce, ts })
    │
    ▼
[4] MQTT Publish via MQTTManager
    Priority: Cloud MQTT → Local MQTT → HTTP → Bluetooth → Offline Queue
    Topic: luma/devices/{id}/commands
    Payload: { action, params, nonce, ts, signature? }
    QoS: 1 (at-least-once)
    │
    ▼
[5] Await Response
    Subscribe to luma/devices/{id}/response (timeout: 5s)
    ├── Success → confirm UI update, audit log
    ├── Timeout → revert optimistic update, push notification
    └── Error from device → revert, show error, audit log
    │
    ▼
[6] Cloud Sync
    POST /cloud/sync → record new device state version
    │
    ▼
[7] Audit Log
    POST /cloud/audit → { actor, action, resource, ip }
```

### 7.2 Action Catalogue

| Action | Target Devices | Params | Permission Required |
|---|---|---|---|
| `TOGGLE` | All | `{ on: boolean }` | `member` |
| `SET_BRIGHTNESS` | Lamp, Dimmer | `{ brightness: 0–100 }` | `member` |
| `SET_COLOR` | RGBW Lamp | `{ r, g, b }` | `member` |
| `SET_COLOR_TEMP` | Tunable Lamp | `{ kelvin: 2700–6500 }` | `member` |
| `SET_FAN_SPEED` | Fan | `{ speed: 0–5 }` | `member` |
| `SET_LOCK` | Smart Lock | `{ locked: boolean }` | `admin` |
| `PULSE` | Any GPIO | `{ durationMs }` | `member` |
| `BLINK` | Any GPIO | `{ count, intervalMs }` | `member` |
| `TIMER_SET` | Any | `{ durationMs, action }` | `member` |
| `TIMER_CANCEL` | Any | — | `member` |
| `SCHEDULE_PUSH` | Any | `{ schedule: Schedule }` | `admin` |
| `SCHEDULE_DELETE` | Any | `{ scheduleId }` | `admin` |
| `FIRMWARE_UPDATE` | Any | `{ version, url, checksum }` | `admin` |
| `FACTORY_RESET` | Any | — | `owner` |

## 10. MQTT Topic Architecture (Unified)

### 10.1 Canonical Topic Hierarchy

All topics adopt the `luma/` namespace prefix. Every topic is scoped by a resource ID so ACLs can be applied at the broker level.

```
luma/
├── users/{userId}/
│   └── presence                   (retained, online/offline)
│
├── microcontrollers/{mcId}/
│   ├── heartbeat                  (QoS 0, every 30s)
│   ├── status                     (retained, QoS 1, LWT target)
│   └── config                     (retained, QoS 1, config push)
│
├── devices/{deviceId}/
│   ├── state        (device → cloud, retained, QoS 1)  ← current truth
│   ├── telemetry    (device → cloud, QoS 0)            ← sensor stream
│   ├── commands     (cloud → device, QoS 1)            ← action requests
│   ├── response     (device → cloud, QoS 1)            ← ack/result
│   ├── status       (device → cloud, retained, LWT)    ← online/offline
│   ├── heartbeat    (device → cloud, QoS 0)            ← keepalive
│   ├── schedule     (cloud → device, retained, QoS 1)  ← schedule push
│   ├── firmware     (cloud → device, QoS 1)            ← OTA notification
│   ├── logs         (device → cloud, QoS 0)            ← debug logs
│   └── events       (device → cloud, QoS 1)            ← alarm/alert events
│
└── system/
    ├── broadcast    (cloud → all, QoS 0)               ← system messages
    └── discovery    (device → cloud, QoS 0)            ← mDNS/broadcast
```

### 10.2 Topic Resolution Rule

**Single source of truth:** All topic strings must be generated by a shared `topicFor(deviceId, kind)` function. No hand-rolled topic strings anywhere in mobile or backend.

```typescript
// Mobile (TypeScript) — update MQTTTopics.ts
const NAMESPACE = "luma";

export type TopicKind =
  "state" | "telemetry" | "commands" | "response" | "status" |
  "heartbeat" | "schedule" | "firmware" | "logs" | "events";

export function topicFor(deviceId: string, kind: TopicKind): string {
  return `${NAMESPACE}/devices/${deviceId}/${kind}`;
}
```

```go
// Backend (Go) — already correct in topics.go, just extend
func TopicsForDevice(deviceID string) Topics {
    base := fmt.Sprintf("luma/devices/%s", deviceID)
    return Topics{
        State:     base + "/state",
        Telemetry: base + "/telemetry",
        Commands:  base + "/commands",
        Response:  base + "/response",
        Status:    base + "/status",
        Heartbeat: base + "/heartbeat",
        Schedule:  base + "/schedule",
        Firmware:  base + "/firmware",
        Logs:      base + "/logs",
        Events:    base + "/events",
    }
}
```

### 10.3 QoS Policy

| Topic Suffix | Direction | QoS | Retained | Notes |
|---|---|---|---|---|
| `/state` | device → cloud | 1 | Yes | Current truth — must survive reconnect |
| `/telemetry` | device → cloud | 0 | No | High frequency, loss acceptable |
| `/commands` | cloud → device | 1 | No | Must deliver; device acks via /response |
| `/response` | device → cloud | 1 | No | Ack for a specific command nonce |
| `/status` | device → cloud | 1 | Yes | LWT: `{"online":false}` |
| `/heartbeat` | device → cloud | 0 | No | 30s interval keepalive |
| `/schedule` | cloud → device | 1 | Yes | Retained so device gets it on reconnect |
| `/firmware` | cloud → device | 1 | No | OTA trigger |
| `/logs` | device → cloud | 0 | No | Debug only |
| `/events` | device → cloud | 1 | No | Alarms, alerts, anomalies |

### 10.4 Last Will & Testament

Every device connection must set LWT at connect time:
```
LWT Topic:   luma/devices/{deviceId}/status
LWT Payload: {"online": false, "reason": "unexpected_disconnect", "ts": 0}
LWT QoS:     1
LWT Retain:  true
```

### 10.5 MQTT Credential Flow

```
Mobile App Login
    │
    ├──► POST /cloud/v1/auth/login → { accessToken, refreshToken }
    │
    ├──► GET  /cloud/v1/devices          → list of owned devices
    │
    └──► For each deviceId:
         POST /cloud/v1/devices/{deviceId}/mqtt-credentials
         ← { clientId, username, password, brokerUrl, expiresAt }
         │
         └──► MQTTManager.connectAll({ cloud: { host, port, username, password } })
```

Credentials are scoped: a phone's MQTT identity can only publish to `luma/devices/{id}/commands` for devices it owns or has admin access to, and can only subscribe to the state/telemetry/response topics of those same devices. The broker (EMQX) enforces this via ACL rules provisioned by the backend's MQTT adapter.

## 11. Security Model

### 11.1 JWT Lifecycle

```
accessToken   → short-lived (15 min), signed RS256 or HS256
refreshToken  → opaque (256-bit random), hashed in DB, 30-day TTL
              → rotated on every use (refresh token rotation)
              → stored in sessions table with phone_id binding
```

Blacklist: Redis-backed in-memory LRU. JTI (JWT ID) added to blacklist on logout/revoke. Token check: `SELECT 1 FROM blacklist WHERE jti = $1` on every authenticated request.

### 11.2 MQTT Authorization

```
Device identity (ESP32):
  - client_id:  luma-device-{deviceId}
  - username:   device-{deviceId}
  - password:   scoped secret (SHA-256 keyed hash), rotated quarterly
  - ACL: publish   luma/devices/{deviceId}/state
               luma/devices/{deviceId}/telemetry
               luma/devices/{deviceId}/response
               luma/devices/{deviceId}/status
               luma/devices/{deviceId}/heartbeat
               luma/devices/{deviceId}/logs
               luma/devices/{deviceId}/events
         subscribe luma/devices/{deviceId}/commands
                   luma/devices/{deviceId}/schedule
                   luma/devices/{deviceId}/firmware

Mobile App identity (per user-phone):
  - client_id: luma-phone-{userId}-{phoneId}-{random}
  - username: user-{userId}
  - password: scoped credential from /mqtt-credentials endpoint
  - ACL: subscribe luma/devices/{id}/state   (for each authorized device)
                   luma/devices/{id}/status
                   luma/devices/{id}/telemetry
                   luma/devices/{id}/response
                   luma/devices/{id}/events
         publish   luma/devices/{id}/commands (for each authorized device)
```

### 11.3 Command Signing

Every command published by the mobile app should include a signature to prevent replay attacks. The `MQTTSecurity.ts` module already implements this:

```typescript
payload = {
  device: deviceId,
  action: command,
  params: params,
  nonce: crypto.randomUUID(),     // unique per message
  ts: Date.now(),                 // unix ms
  signature: SHA256_HMAC(payload_without_sig, deviceKey)
}
```

Devices verify `ts` is within ±60 seconds (replay window) and `nonce` has not been seen before (recent nonce cache in device firmware).

### 11.4 Permission Hierarchy

```
owner
  └── admin (delegated subset of owner powers)
       └── member (can control authorized devices)
            └── guest (time-limited, device-limited access)
```

Permission check order:
1. Is user `owner` of the microcontroller? → full access
2. Is user in `device_admins` for this device? → admin access
3. Is user in `permissions` table for this device? → role-based access
4. MCU-level `mcUsers` entry with `deviceAccess` feature list? → granular feature access
5. Default: deny

### 11.5 Secure Local Storage

Secrets (JWT tokens, MQTT passwords, device keys) must never be stored in `AsyncStorage` (plaintext). Use:
- iOS: Keychain via `expo-secure-store`
- Android: Android Keystore via `expo-secure-store`

## 12. Synchronization Model

### 12.1 Sync Strategy

Use **optimistic local-first** sync with version vectors:

```
Each resource type has a `version` integer in the DB.
Each phone tracks `last_synced_version` per resource type in `sync_states`.

On app foreground:
  GET /cloud/v1/sync/delta?since={last_synced_version}&types=devices,scenes,schedules
  ← { changes: [{ resourceType, resourceId, version, action, data }] }
  Apply changes to local state.
  Update last_synced_version.

On local mutation:
  Apply change locally (optimistic).
  POST /cloud/v1/sync/push → { resourceType, resourceId, version, data }
  ← { accepted: true, version: N }  OR  { conflict: true, serverData: {...} }
```

### 12.2 Conflict Resolution

| Strategy | When to Apply |
|---|---|
| **Last-write-wins** | Device state (on/off, brightness) — real-time, always use latest |
| **Server-wins** | Permissions, user roles — authoritative on server |
| **User-resolves** | Scenes, automation rules — present diff to user |
| **Merge** | Schedules — merge non-overlapping additions |

### 12.3 Offline-First Data Priority

```
1. Local in-memory state (LumaContext)  ← fastest, always available
2. Local DB (DatabaseEngine / AsyncStorage)  ← survives app restart
3. Cloud sync (REST API)  ← canonical truth
4. MQTT retained messages  ← device truth
```

On reconnection after offline period:
1. Drain offline MQTT command queue
2. Pull sync delta from cloud
3. Reconcile conflicts
4. Re-subscribe to device topics

### 12.4 Multi-Device (Multi-Phone) Sync

When the same user has two phones (or phone + tablet):
- Both subscribe to the same device MQTT topics → receive same state updates
- Mutations on one phone: pushed to cloud → cloud broadcasts via push notification (FCM/APNs) to other phones → other phones pull delta
